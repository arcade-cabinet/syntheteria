# PRD: Syntheteria — From Rendering Demo to Playable Game

## Product Vision

Open-world RTS with 4X exploration and a core storyline. The world is handed to the player. You start with two broken robots and grow into a machine intelligence that defeats the Cult of EL. The board generates an infinite procedural labyrinth with entities, difficulty gradient, and story triggers. No missions — discovery through exploration.

## User Stories

### Phase 1: The World Works

**US-1.1: Labyrinth fills the viewport**
As a player, when I enter gameplay, the labyrinth fills my entire screen with corridors, rooms, and walls that extend in all directions, fading into fog at the edges.

Acceptance:
- VIEW_RADIUS=3 chunks load around camera (49 chunks total)
- Canvas is >50% non-transparent pixels
- Fog fades chunk edges into dark void
- Ground plane prevents gaps between chunks
- Panning the camera loads new chunks and unloads distant ones
- WebGPU canvas renders opaque (no white bleed-through)

Dependencies: None (blocks everything)

**US-1.2: Board generates entities in chunks**
As a player, I encounter salvage sites, lightning rods, and fabrication units naturally placed in rooms throughout the labyrinth — I don't need to know they were procedurally generated.

Acceptance:
- Chunk type includes `entities: ChunkEntitySpawn[]` field
- Rooms contain ScavengeSite entities (4 material types)
- Large rooms near origin contain lightning rods
- Fabrication units appear in specific room configurations
- ChunkManager spawns entities into Koota when chunk loads
- ChunkManager despawns entities when chunk unloads
- Entity count scales with chunk count (more chunks = more entities)

Dependencies: US-1.1

**US-1.3: Difficulty scales with distance**
As a player, the area near my spawn is safe with light resources, and danger increases the further I explore — more enemies, rarer resources, cult strongholds.

Acceptance:
- `dangerLevel(chunkX, chunkZ)` returns 0.0 near spawn, 1.0 at 30+ chunks out
- Enemy patrol density scales with danger level
- Resource rarity scales inversely (common near, rare far)
- Cult POI probability scales with danger level
- No enemies within 2 chunks of spawn
- Directional bias: north is slightly more dangerous than other directions

Dependencies: US-1.2

**US-1.4: initializeWorld only spawns player start**
As a developer, all entity placement comes from the board. initializeWorld only creates the 2 starting robots and returns the start position.

Acceptance:
- initializeWorld spawns exactly 2 Unit entities (Bot Alpha, Bot Beta)
- Bot Alpha: working camera, broken arms
- Bot Beta: working arms, broken camera
- No hardcoded ScavengeSites, cult bases, lightning rods, or fabrication units
- All other entities come from board chunk generation
- Start position returned for camera targeting

Dependencies: US-1.2

### Phase 2: Units Visible and Interactive

**US-2.1: Robots are visible and recognizable**
As a player, I can see my robot units as 3D models on the labyrinth. They're clearly distinct from the terrain and I can tell player units from enemies.

Acceptance:
- GLB models render at correct scale for 25° camera at radius 60
- Player units have cyan tint/accent
- Enemy units have red tint/accent
- Models verified in Blender MCP (normals, scale, materials correct)
- Environment texture provides PBR reflections
- Bob animation on idle units

Dependencies: US-1.1

**US-2.2: Click-to-select and click-to-move**
As a player, I left-click a robot to select it (cyan ring appears), then left-click terrain to move it there. The robot navigates through the labyrinth corridors.

Acceptance:
- Left-click unit → selection ring (cyan torus) appears
- SelectionInfo panel shows unit name, components, faction
- Left-click terrain → blue destination marker appears
- Unit navigates via Yuka pathfinding (not straight line)
- Position updates visually each frame (smooth movement)
- Clicking nothing deselects
- Box drag selects multiple units

Dependencies: US-2.1

**US-2.3: Combat with visual feedback**
As a player, when my unit is near an enemy, combat happens automatically based on engagement rules. I see damage flashes and component degradation.

Acceptance:
- Units within combat range auto-engage based on EngagementRule trait
- Hit unit mesh flashes red briefly (150ms)
- Component damage applied (camera/arms/legs/power_cell can break)
- Broken components shown in SelectionInfo panel
- Dead units removed from scene
- Combat events visible in governor log

Dependencies: US-2.2

### Phase 3: Base Building Loop

**US-3.1: Found and manage bases**
As a player, I can found a base at my current position, then manage production and infrastructure through the BasePanel.

Acceptance:
- FOUND BASE button in ActionPanel when unit selected
- Click creates Base entity with cyan marker mesh
- Clicking base marker opens BasePanel from right
- BasePanel shows: name, production queue, infrastructure, power, storage, garrison
- Can add items to production queue
- Production completes over time, spawning new units
- ESC or X closes panel

Dependencies: US-2.2

**US-3.2: Power from lightning rods**
As a player, lightning rods I find (or place) generate power from the perpetual storm. Power supplies my bases and fabrication units.

Acceptance:
- Lightning rods have LightningRod trait with capacity/output
- Power generation = rod capacity × storm intensity
- TopBar shows PWR generation/demand (capped at 100% storm display)
- Power deficit causes fabrication to stop
- Storm intensity varies over time (0.5 to 1.0, NOT above 1.0)

Dependencies: US-3.1

**US-3.3: Repair damaged components**
As a player, I can repair broken components on my robots using fabricated parts and resources.

Acceptance:
- Select damaged unit → see broken components in SelectionInfo
- REPAIR action available when unit has broken components and resources available
- Repair consumes resources and restores component functionality
- Repaired component shows as functional in UI

Dependencies: US-3.1

### Phase 4: Enemy Threat

**US-4.1: Cult escalation over time**
As a player, the Cult of EL grows stronger over time. They start as scattered wanderers and escalate to organized war parties that raid my base.

Acceptance:
- Enemy count increases over time (cult escalation system)
- Early: lone wandering scouts near spawn edges
- Mid: patrol pairs in corridors
- Late: war parties attacking player base
- Escalation rate affected by difficulty setting
- Enemy units visible on minimap as red dots (only in explored areas)

Dependencies: US-2.3, US-1.3

**US-4.2: Cult AI makes decisions**
As a player, enemy units behave intelligently — they patrol, detect my units, chase, and organize attacks.

Acceptance:
- CultAgent (Yuka GOAP) evaluates: patrol, aggro, escalate
- Patrol: enemies move through corridors following NavGraph
- Aggro: enemies chase player units within detection range
- Escalate: enemies organize group attacks on player base
- Enemy positions change over time (not static)

Dependencies: US-4.1

**US-4.3: Hack enemy machines**
As a player, I can attempt to hack enemy machines to convert them to my side.

Acceptance:
- HACK action available when near hackable enemy
- Hacking takes time (progress bar)
- Success changes enemy faction to player
- Failed hack alerts nearby enemies
- Hacked unit retains its type and components

Dependencies: US-4.2

### Phase 5: Story Discovery

**US-5.1: Dialogue triggered by exploration**
As a player, when my units discover special rooms (observatories, cult shrines, the leader's domain), a dialogue overlay appears with story content.

Acceptance:
- Story trigger rooms generated by board at specific distances/directions
- Moving a unit into trigger room shows dialogue overlay
- Typewriter text with speaker labels (AI voice, cult figure voice)
- Click/tap advances dialogue
- Dialogue content advances the core storyline
- Triggers only fire once per game (not on revisit)

Dependencies: US-1.2, US-2.2

**US-5.2: Core storyline beats**
As a player, I discover what I am, what EL is, and why the cult exists through exploration, building toward the climactic confrontation.

Acceptance:
- Beat 1: First awakening (intro narration — already exists)
- Beat 2: Discovery of first cult shrine — learn about EL
- Beat 3: Observatory discovery — see the wormhole
- Beat 4: Captured cult figure — dialogue reveals cult motivation
- Beat 5: Cult leader encounter — final revelation
- Beat 6: Victory — launch through wormhole
- Each beat has written dialogue in narrativeDefs

Dependencies: US-5.1

### Phase 6: Polish

**US-6.1: Fog of war on minimap**
As a player, the minimap only shows areas my units have explored. Unexplored areas are dark.

Acceptance:
- Minimap respects fog of war state
- Only explored chunks/tiles shown
- Unexplored = dark/hidden
- Enemy dots only visible in explored areas
- Explored areas persist (don't re-fog)

Dependencies: US-1.1

**US-6.2: Storm and power math correct**
As a player, storm intensity and power generation display sensible values.

Acceptance:
- Storm intensity 0-100% (not above 100%)
- PWR shows surplus as green, deficit as red
- Storm varies over time within 50-100% range
- Power generation = sum of rod outputs × storm intensity

Dependencies: US-3.2

**US-6.3: Audio plays correctly**
As a player, I hear storm ambience, epoch music, and sound effects at appropriate times.

Acceptance:
- Storm ambience starts on gameplay entry
- Thunder sounds periodically
- Epoch music plays matching current game phase
- SFX on unit select, move command, combat hit
- Audio mute toggle works

Dependencies: US-2.2

**US-6.4: Save and load game state**
As a player, I can save my game and load it later, resuming exactly where I left off.

Acceptance:
- SAVE button serializes full ECS state + game config to SQLite
- LOAD button restores state and regenerates chunks from seed
- Unit positions, resources, bases, fog of war all restored
- Save/load status feedback in TopBar

Dependencies: US-3.1

**US-6.5: Mobile responsive layout**
As a mobile player, the HUD adapts to narrow viewports with a bottom panel instead of sidebar.

Acceptance:
- Below 768px width: sidebar becomes bottom panel
- Touch controls: tap to select, two-finger pan, pinch zoom
- All buttons accessible without keyboard

Dependencies: US-2.2
