# Test Plan — Syntheteria Visual & Interaction Testing

## Philosophy

- **No mocks** unless genuinely necessary (e.g., WebGL context in jsdom). If a test needs mocks to work, the test is wrong or the component has too many dependencies.
- **Component tests** render real React components in real Chrome (vitest browser plugin, headed). They test what a PLAYER sees and does — clicks, text content, visual states.
- **E2E playthrough** navigates the real app from landing page through gameplay with the PlaytestGovernor making player decisions. No forced spawns, no shortcuts. The game plays itself like a player would.
- **Unit tests** (existing 725 in jsdom) cover pure logic — combat math, pathfinding, resource calculations. These don't change.

### Tier 2: Full E2E Playthrough (vitest browser, headed Chrome)

#### 2.1 Complete Game Playthrough (`tests/components/playthrough.browser.test.tsx`)

This is THE test. It plays the game from start to finish as a player would, using the PlaytestGovernor for automated decisions.

**Flow:**
1. Render `<App havok={null} />` — landing screen appears
2. Click [ NEW GAME ] button
3. Verify modal opens, click [ START ]
4. Verify narration overlay appears, click SKIP
5. Verify gameplay HUD renders (TopBar, Sidebar with "No Selection")
6. Enable governor via `enableAutoPlay()`
7. Force-tick 50 simulation ticks
8. Verify: tick counter advanced, governor made decisions, units exist
9. Force-tick 50 more ticks
10. Verify: resources changed, game phase may have advanced
11. Check governor action types (explore, attack, scavenge, found_base)
12. Check zero console errors (non-engine)

**Key assertions:**
- Phase transitions: title → narration → playing (verified by DOM content)
- Governor produces actions (getGovernorLog().length > 0)
- ECS state advances (tick > 0, units exist)
- No uncaught exceptions

**Why this works without BabylonJS canvas:**
The test renders `<App>` which includes `<GameCanvas>`. In the vitest browser environment (Vite-based, not Webpack), Reactylon components won't compile (needs babel-plugin-reactylon). So `<GameCanvas>` will error. The test should catch this gracefully — the HUD and ECS still work, just no 3D canvas. The playthrough tests the React DOM layer + ECS game loop, which is what matters for automated testing. Visual verification of the 3D canvas is done manually via Chrome DevTools MCP.

**Alternative:** If GameCanvas import fails, mock ONLY that one component:
```typescript
vi.mock("../../src/game/GameCanvas", () => ({
  GameCanvas: () => <div data-testid="game-canvas-stub" />
}));
```
This is the ONE acceptable mock — BabylonJS/Reactylon genuinely can't run in the vitest Vite environment without babel-plugin-reactylon.

## Rules for Writing These Tests

1. **NO `vi.mock()`** except the one GameCanvas stub in the playthrough test
2. **Real ECS state** — spawn entities with `world.spawn()`, query with `world.query()`
3. **Real React rendering** — `createRoot(container).render(<Component />)`
4. **Real user actions** — `element.click()`, DOM queries for text content
5. **Clean up after each test** — `afterEach(cleanup)` unmounts React, destroy spawned entities
6. **Test what the PLAYER sees** — text content, button states, visual presence — not implementation details
7. **Test how things BREAK** — disabled buttons, missing entities, destroyed entities, empty state
