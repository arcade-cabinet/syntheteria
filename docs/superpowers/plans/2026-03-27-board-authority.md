# Board Authority — src/board as Central Agency

## The Problem

src/board generates terrain but doesn't place entities. All entity spawning is hardcoded in `initializeWorld.ts`:
- Player units at fixed positions
- Cult bases at hardcoded tile coordinates
- Scavenge sites manually spawned around player start
- Zones constrained to fixed `WORLD_EXTENT = 256` (finite world, not infinite)

This means the "infinite chunk-based world" is actually just terrain — no procedural content placement, no emergent world, no zone-driven encounters.

## The Solution

### src/board becomes the bridge between Koota ECS and Reactylon rendering

When a chunk is generated, the board determines:
1. **Terrain** (walls, floors, rooms, corridors) — already works
2. **Salvage sites** — placed in rooms based on zone and noise
3. **Enemy patrol points** — placed based on distance from player start
4. **Cult POIs** (bases, shrines, outposts) — placed in enemy territory chunks
5. **Event triggers** — zone transitions, discovery points
6. **Resource density** — varies by zone/floor

### Zones as Floors, Not Fixed Boundaries

Instead of `WORLD_EXTENT = 256` with quadrant zones on a flat plane:

- **Floor 0 (City):** The starting area. Industrial labyrinth. Moderate salvage, few enemies.
- **Floor 1 (Coast):** Accessed via portal. Coastal mines, more resources, scattered cultists.
- **Floor 2 (Campus):** Science campus floor. Research POIs, observatory, story elements.
- **Floor 3 (Enemy Territory):** Cult strongholds, heavy resistance, leader's domain.

Each floor is its own infinite chunk grid with its own seed derivative (`${worldSeed}_floor${n}`). Portals are special rooms generated by the chunk pipeline that connect floors.

### Chunk Generation Pipeline (Updated)

Current (9 phases — terrain only):
1. Room placement
2. Maze fill
3. Border gates
4. Region connectivity
5. Dead-end pruning
6. Re-open gates
7. Zone floor assignment
8. Resource scatter
9. Zone stamping

Proposed (12 phases — terrain + entity placement):
1-9. Same as current
10. **Entity placement** — based on zone/floor:
    - Salvage sites in rooms (density from zone profile)
    - Enemy spawn points in corridors (probability from floor danger level)
    - POI markers (cult bases, shrines) in large rooms on enemy floors
11. **Portal placement** — in specific room types, connecting to other floors
12. **Event trigger placement** — discovery zones, story triggers

### What initializeWorld Does (After Refactor)

Only:
1. Set game config (seed, difficulty)
2. Generate the initial board layout
3. Build nav graph
4. Spawn the 2 starting player robots (the "you wake up" moment)
5. Run initial simulation tick

Everything else — salvage, enemies, bases, POIs — is placed by the board during chunk generation.

### Implementation Order

1. Add entity spawn data to `Chunk` type (new field: `entities: ChunkEntitySpawn[]`)
2. Add phase 10-12 to chunk generation pipeline
3. Wire chunk entity spawns into ECS via `ChunkManager` (when chunk loads, spawn entities; when unloads, despawn)
4. Remove hardcoded spawns from `initializeWorld.ts`
5. Replace `WORLD_EXTENT` zone system with floor-based zones
6. Add portal room generation
7. Fix storm/power math (cap storm display at 100%, fix power balance)
8. Fix minimap to respect fog of war
9. Fix robot GLBs via Blender MCP

### Math Fixes

**Storm 112%:** `stormIntensity` ranges 0.0-1.5 but displays as percentage. Either cap at 1.0 or display as "STORM: SEVERE" text tiers instead of percentage.

**PWR 11/9 (generation > demand):** This may actually be correct — surplus power is good. But display should be `PWR +2 surplus` or `PWR 11/9 ▲` to indicate healthy state vs deficit.
